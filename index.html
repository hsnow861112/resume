프로젝트

Zenius APM (Web Application Server 모니터링)

Java를 기반으로 한 Web Application Server의 성능, 장애에 대한 모니터링 기능을 제공 하는 솔루션.
- 사용 기술
* JavaScript, Jquery, Jquery UI, JSP, Spring, MyBatis, CanvasJS, PostgreSQL
- 인력 구성
* 개발자: 4명 , 디자인: 1 명
- 상세 업무
1. LOWCODE화 시킨 사용자 정의 대시보드
미리 형태가 정해져 있는 대시보드가 아닌 마치 파워 포인트 와 같이 성능 항목들을 drag & drop으로
사용자가 원하는 위치에 설정하고 또한 위치 시킨 컴포넌트들의 width, height 도 drag 로 설정 할 수
있게 설계 하였고 이러한 기능들을 구현하기 위해서 jQuery UI의 Draggable, Droppable, Resizable을
기술 조사하여 개발함.


* 구현 예제 (오른쪽 항목을 모자이크 영역으로 drag & drop 및 우측 하단 영역을 드래그해서 resize)
2. 스피드 바 차트 개발
RPS(초당 요청 건수), Active Transaction(현재 처리중인 건수), TPS(초당 처리 건수) 를 나타내는 차트를
html5 Canvas 및 애니메이션을 위한 반복함수 requestAnimationFrame을 기술 조사하여 차트를 개발함.

* 구현 예제 (실시간 요청, 처리가 발생 했을때 애니메이션 효과.)
* 문제 해결 사례
- 실시간 요청, 처리가 발생 했을때 애니메이션 효과 구현 방법은 다음과 같습니다.

1. 1번 직선의 첫번째 점 x1을 점점 증가시키면서 직선을 왼쪽에서 오른쪽으로 이동시킴.
2. 이동 도중 그려야 하는 직선의 길이(width) 안에 고정된 커브 영역 좌표 x2가 포함되어있으면 직선을 x1 ~ x2까지만 그림.
3. 커브 애니메이션을 구현하기 위해 고정된 커브 영역좌표에서 선의 마지막 좌표로 향하는 직선을 그림.
4. 2번 직선의 우측 마지막 x좌표는 x2 + (직선의 길이(width) – (x2 – x1)).
5. y값을 구하기 위해 선의 마지막 좌표와 고정된 커브 영역 좌표 두개를 활용해서 기울기를 구하고
((y3 – y2) / (x3 – x2)) 일차함수 y = ax + b 에서 a에 기울기를 대입 하고 x에 2번 직선의
우측 마지막 x좌표를 대입 해서 b의 값을 구해서 방정식을 완성 시킨 후 y값을 구한 후 고정된 커브
영역 좌표와 2번 직선의 우측 마지막 x좌표를 연결 해서 선을 그림.
* 구현 예제 (실시간 요청이 서버(가운데 원통) 영역으로 들어 갔을때 버블 효과.)
* 문제 해결 사례
- 실시간 요청이 서버(가운데 원통) 영역으로 들어 갔을때 버블 효과 구현 방법은 다음과 같습니다.

1. 한 점에 있는 여러 개의 원이 퍼지는 애니매이션을 구현하기 위해서 각 각도에 따른 좌표를 구해야 했음.
2. 전방위로 퍼져나가야 하기 때문에 360도에서 원의 개수를 나누어서 각도(DEGREE)를 구하고 원이 퍼져 나갈 마지막 길이(R)를 정의함.
3. 길이(R)과 각도(DEGREE)를 안다면 삼각함수를 사용해서 SIN(degree) = Y / R, COS(DEGREE) = X / R을 구할 수 있고 그에 따라 Y = SIN(DEGREE) * R, X = COS(DEGREE) * R을 유추 할 수 있음
4. 길이(R)을 0에서 점점 증가시킨다음 그 상태에서 X, Y값에 원을 그려서 점점 퍼져 나가는 애니매이션을 구현함. 퍼저나가는 원의 개수 및 크기를 랜덤화 시켜서 좀 더 극적인 효과가 보여지게끔 구현함
3. X-VIEW 차트 개발
현재 WAS가 처리한 트랜잭션을 응답시간, 처리상태 별로 나누어서 표현하는 X-VIEW 차트를 html5 canvas를 사용해서 개발.

* 구현 예제 ( 데이터 생성 및 삭제는 Web Worker를 통해 이루어지고 있습니다. )
* 문제 해결 사례
- 대용량 점들의 실시간 차트 애니매이션 구현.
대용량으로 들어오는 트랜잭션 건수를 점으로 변환 해서 시간이 흐름에 따라 점이 점점 왼쪽으로 흐르게 했어야 했는데
html5 canvas 특성 상 전부 지웠다가 다시 그리게 했어야 했습니다. 만약 트랙잭션이 몇 개 안된다면 다시 그리게 해서
실시간 차트 애니매이션을 만드는건 어렵지 않지만 초당 처리 건수가 3000개가 넘어가는 경우, 차트의 시간 축이 5분이라면
90만개의 점을 초마다 지우고 다시 그리게 된다면 엄청 큰 성능저하의 원인이 되었습니다.
그래서 아래 그림과 같이 왼쪽으로 이동 할 때 조회되면 안되는 영역의 x축 크기를 구한 다음 해당 x축 크기부터 x축의
마지막 좌표까지 캡쳐 한 후 다음 번에 그릴 때 canvas영역에 그려져 있는걸 모두 지우고 캡쳐한 이미지를 붙혀 넣는
방식으로 해결 할 수 있었습니다.
