1. 실시간 대시보드 개발

WAS에 인입되고 처리되는 모든 트랜잭션들을 실시간으로 모니터링하고, 지연 상황을 가시적으로 분석 할 수 있는 실시간 대시보드 개발.

문제 해결 사례

1. 브라우저 Memory Leak 문제 해결 경험

[문제 설명]
대시보드를 하루 종일 실행시켜 놓고 모니터링해야 하는데 특정 시간이 지나면 브라우저가 항상 Memory Leak로 인한 비정상적으로 종료됨.

[해결 과정]
크롬 개발자 도구 > 메모리 탭 > 힙 스냅샷을 시간 마다 촬영하여 shallow size, retained size 를 통하여 x-view 차트에서 data가 제대로 지워지지 않는 것을 확인. 삭제 관련 로직 중 date 부등호가 잘못 되어있어 이부분 개선하여 문제를 해결함.

2. 대시보드 화면에서 다른 화면으로 이동 후에도 Memory가 증가하는 현상 해결.

[문제 설명]
대시보드 화면에서 다른 화면으로 이동했지만 아무 이유 없이 브라우저 속도가 느려지는 문제가 발생.

[해결 과정]
화면 이동으로 실시간 트랜잭션 data가 필요 없어졌음에도 불구하고 WebWorker를 제대로 종료하지 않아 Memory가 계속증가하여 문제 발생하는 것 확인. WebWorker를 통해 data가 들어 왔을때 현재 DOM에 x-view 차트의 root tag를 찾아서 없다면 종료 하는 방식으로 문제를 해결.

3. WebWorker를 사용한 UI 성능 개선.

[문제 설명]
WebSocket을 통해 초단위로 데이터를 받아서 가공을 하고 있었는데 트랜잭션이 많아져서 순간적으로 가공한 데이터의 개수가 많아지면 다른 실시간 차트들의 애니매이션 및 UI 성능이 저하되서 품질이 낮아지는 문제가 발생 했었음.

[해결 과정]
JavaScript의 메인 쓰레드에서 대량의 데이터를 처리할 때 다른 차트의 애니메이션 및 UI 성능에 영향을 미치는 문제는 여전히 해결되지 않았음. 그래서 백그라운드 쓰레드인 WebWorker를 활용 하기로 하여 처음에는 WebSocket의 onmessage 이벤트를 통해 받은 데이터를 Web Worker로 직접 전송하여 가공하기로함. 하지만 이렇게 하더라도 대용량의 데이터를 WebWorker로 전송하는 것 자체가 성능 저하를 야기했기 때문에 UI 성능에는 큰 변화가 없었음. 결국에는, WebSocket 자체를 메인 쓰레드가 아닌 WebWorker 내부에 설정하고 데이터를 가공하는 방식으로 변경하여 문제 해결함.

현재 WAS가 처리한 트랜잭션을 응답시간, 처리상태 별로 나누어서 표현하는 X-VIEW 차트를 html5 canvas를 사용해서 개발.

3. 스피드 바 차트 개발

RPS(초당 요청 건수), Active Transaction(현재 처리중인 건수), TPS(초당 처리 건수) 를 나타내는 차트를 html5 Canvas 및 애니메이션을 위한 반복함수 requestAnimationFrame을 기술 조사하여 차트를 개발함.

구현 예제

1. 상단의 스피드 바 차트 사진에서 왼쪽(초당 요청 건수), 오른쪽(초당 처리 건수) 영역의 애니메이션.

2. 실시간 요청이 서버(가운데 원통) 영역으로 들어 갔을때 버블 효과.

문제 해결 사례

- 스피드바 차트 애니매이션 구현.

[문제 설명]
APM 대시보드의 상징적인 차트인 스피드바를 제공하는 라이브러리가 없어서 직접 구현함.

[해결 과정 - 1]
RPS, TPS를 나타내는 선이 원통영역으로 날라갈때 커브 애니메이션의 구현.

1. 1번 직선의 첫번째 점 x1을 점점 증가시키면서 직선을 왼쪽에서 오른쪽으로 이동시킴.

2. 이동 중 그려야 하는 직선의 길이(width) 안에 고정된 커브 영역 좌표 x2가 포함되어있으면 직선을 x1 ~ x2까지만 그림.

3. 커브 애니메이션을 구현하기 위해 고정된 커브 영역좌표에서 선의 마지막 좌표로 향하는 직선을 그림.

4. 2번 직선의 우측 마지막 x좌표는 x2 + (직선의 길이(width) – (x2 – x1)).

5. y값을 구하기 위해 선의 마지막 좌표와 고정된 커브 영역 좌표 두개를 활용해서 기울기를 구하고 ((y3 – y2) / (x3 – x2)) 일차함수 y = ax + b 에서 a에 기울기를 대입 하고 x에 2번 직선의 우측 마지막 x좌표를 대입 해서 b의 값을 구해서 방정식을 완성 시킨 후 y값을 구한 후 고정된 커브 영역 좌표와 2번 직선의 우측 마지막 x좌표를 연결 해서 선을 그림.

[해결 과정 - 2]
두번째 선이 원통영역으로 다 날라가고 나서 물방울이 터지는 애니매이션 효과 구현.

1. 한 점에 있는 여러 개의 원이 퍼지는 애니매이션을 구현하기 위해서 각 각도에 따른 좌표가 필요.

2. 전방위로 퍼져나가야 하기 때문에 360도에서 원의 개수를 나누어서 각도(DEGREE)를 구하고 원이 퍼져 나갈 마지막 길이(R)를 정의함.

3. 길이(R)과 각도(DEGREE)를 안다면 삼각함수를 사용해서 SIN(degree) = Y / R, COS(DEGREE) = X / R을 구할 수 있고 그에 따라 Y = SIN(DEGREE) * R, X = COS(DEGREE) * R을 유추 할 수 있음.

4. 길이(R)을 0에서 점점 증가시킨다음 그 상태에서 X, Y값에 원을 그려서 점점 퍼져 나가는 애니매이션 구현 + 퍼저나가는 원의 개수 및 크기를 랜덤화 시켜서 좀 더 극적인 효과가 보여지게 구현.

4. LOWCODE화 시킨 사용자 정의 대시보드

대시보드에 필요 하는 컴포넌트 항목을 drag & drop으로 사용자가 원하는 곳에 위치 시키고 컴포넌트들의 가로, 세로 크기도 drag 로 조절 가능하도록 설계 하고 이를 jQuery UI의 Draggable, Droppable, Resizable을 기술을 통하여 구현함.

WebWorker 안에서는 메인 쓰레드에 import 되어있는 diff.js를 사용 할 수 없기 때문에 따로 script를 import 할 수 있는 방법을 찾아야 했고 https://developer.mozilla.org/en-US/docs/Web/API/WorkerGlobalScope/importScripts 해당 문서에서 그 방법을 찾아 해당 라이브러리를 사용할 수 있었음. WebWorker는 기본적으로 외부 js에 onmessage를 구현해야 하는데 유지 보수할 때 Vue 프로젝트와 Web Application에 있는 js를 번갈아 가면서 보면 가독성이 떨어지기 때문에 https://developer.mozilla.org/en-US/docs/Web/API/Web_Workers_API/Using_web_workers 참조해서 Embedded형태의 WebWorker를 사용해서 가독성을 좋게했음.
